# Tier 3 Submission Guide

A submission is a directory of exactly four files, packed into one encrypted
archive `t3_NNN_slug.zip` by `tools/pack.py`. All four files count as
protected content: they exist in the repository only inside the archive.

| File | Role |
|---|---|
| `problem.md` | Natural-language problem statement (what an evaluated system sees) |
| `ground_truth.py` | Validator: reads solution JSON from stdin, prints verdict JSON |
| `solution.py` | Example solution proving the problem is well-posed |
| `metadata.yml` | Public-facing metadata (mirrored, minus spoilers, in INDEX.md) |

Start from `template/`, which contains a complete working (but deliberately
easy) example.

## problem.md

- **At most 1,000 words and 12 KB.** Tier 1–2 problems range from 158 to 835
  words (median ≈ 330); Tier 3 statements should be in the same style. The cap
  exists so that instance data is either small or *generated by a compact rule
  stated in the text* (e.g. "vertices 0–99 with an edge (i, i+1) for every i").
  Problems consisting of huge enumerated fact lists are out of scope.
- **Self-contained.** All instance data is in the statement; no external files,
  URLs, or references needed to solve the problem.
- **Unambiguous.** The statement must define precisely what constitutes a
  valid solution; every constraint is stated explicitly. The benchmark tests
  modeling a clearly stated problem in ASP/clingo, not guessing the author's
  intent — difficulty must come from the modeling, never from vagueness.
- **Specifies the output format exactly**: name every required JSON field, its
  type, and its semantics (a short JSON schema or an annotated example, as in
  `template/problem.md`), so that a correct solution in the wrong format is
  essentially impossible.
- For optimization problems, states explicitly that the optimum is required
  and that feasible-but-suboptimal solutions are invalid.
- **Ends with the canary line** printed by `python tools/pack.py canary`.

## ground_truth.py

- **Plain Python 3, standard library only.** No clingo, no third-party imports
  (`pack.py` rejects them). Validation must be independent of any solver.
- Reads the solution as JSON from **stdin**; prints exactly one JSON verdict to
  **stdout**: `{"valid": true/false, "message": "..."}`.
- **Accepts every correct solution.** Problems often have many valid solutions;
  the validator checks the constraints semantically, never string-compares
  against a stored answer.
- **Optimization problems**: define the known optimum as a module-level
  constant `EXPECTED_OPTIMAL = <value>` and reject solutions that do not attain
  it (strict optimality, as in Tiers 1–2). The value must match `optimal_value`
  in `metadata.yml`; `pack.py` cross-checks this. Establish the optimum
  reliably — the planted-solution methodology from the ASP-Bench paper
  (construct constraints around a known solution) is recommended.
- Must finish within **10 seconds**.
- Contains the canary string as a comment.

## solution.py

- Python ≥ 3.10 using the **clingo Python API, version 5.8.x**
  (`pip install clingo==5.8.*`); record the tested version in `metadata.yml`.
  CI validates against a pinned 5.8 release.
- Prints the solution JSON to stdout; `python solution.py | python
  ground_truth.py` must yield `"valid": true`.
- Must finish end-to-end within **60 seconds**; the problem must be designed so
  that each individual clingo solve call stays within **20 seconds** on
  commodity hardware (the ASP-Bench design limit).

## metadata.yml

| Field | Requirement |
|---|---|
| `number` | Integer 1–999: next free number in INDEX.md (maintainer may renumber on merge) |
| `slug` | Lowercase `[a-z0-9_]`; archive is named `t3_NNN_slug.zip` |
| `title` | Short public title (must not spoil the solution) |
| `authors` | List of names, as they should appear in INDEX.md and Zenodo |
| `contact` | Email address for review questions |
| `aspects` | Non-empty subset of `OPT, TEMP, DEFAULT, RESOURCE, RECURSIVE, SPATIAL, QUANT` (see paper, Sec. 3.3) |
| `optimization` | `true`/`false`; if `true`, `optimal_value` is required |
| `optimal_value` | The known optimum (numeric), matching `EXPECTED_OPTIMAL` in the validator |
| `clingo_version` | 5.8.x version string the solution was tested with |
| `license` | Exactly `Apache-2.0` — this line is the contribution license grant |
| `notes` | Optional; free-form remarks to the maintainer |

## What pack.py / CI check

`python tools/pack.py verify DIR` (CI runs the same checks on the decrypted
archive):

1. all four files present; `metadata.yml` fields as specified above;
2. `problem.md` within 1,000 words / 12 KB, mentions JSON, contains the canary;
3. `ground_truth.py` stdlib-only, contains the canary, and (for optimization)
   `EXPECTED_OPTIMAL` matches `optimal_value`;
4. `solution.py` runs in ≤ 60 s, its output is accepted by `ground_truth.py`
   run in ≤ 10 s;
5. computes the plaintext SHA-256 fingerprint recorded in INDEX.md.

Checks 1–5 are necessary, not sufficient: the maintainer additionally reviews
novelty and quality and runs the baseline-agent hardness test (see README).

## Leak policy

Plaintext problem content must never appear anywhere GitHub stores publicly:
files, commits, PR titles/descriptions/comments, issues, or CI output. GitHub
retains commits reachable by hash even after branches are deleted, so leaks
are permanent. A leaked submission is treated as contaminated: it can only be
resubmitted after substantial rework (different instance data at minimum,
ideally a different problem structure). When discussing a submission in a PR,
refer to constraints abstractly ("the third capacity constraint") rather than
quoting the text.

## Practical tips

- Develop under `work/` (gitignored) to make accidental commits of plaintext
  impossible.
- Calibrate difficulty: if a frontier coding agent solves your problem in a
  handful of iterations, it belongs in Tier 2, not Tier 3. Good sources of
  hardness observed in Tiers 1–2: dense interaction of constraints within one
  domain (nonograms, sequence assembly) rather than sheer breadth of aspects.
- Verify your optimum twice: once via your `solution.py`, once by an
  independent argument (exhaustive search on a reduced instance, planted
  construction, or a manual proof).
