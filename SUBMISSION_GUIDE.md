# Tier 3 Submission Guide

A submission is a directory of exactly four files, packed into one encrypted
archive `t3_NNN_slug.zip` by `tools/pack.py`. All four files count as
protected content: they exist in the repository only inside the archive.

| File | Role |
|---|---|
| `problem.md` | Natural-language problem statement (what an evaluated system sees) |
| `ground_truth.py` | Validator: reads solution JSON from stdin, prints verdict JSON |
| `solution.py` | Example solution proving the problem is well-posed |
| `metadata.yml` | Public-facing metadata (mirrored, minus spoilers, in INDEX.md) |

Start from `template/`, which contains a complete working (but deliberately
easy) example.

## Core rule: the solution is a clingo encoding, not a Python search

Tier 3 tests one specific skill — modelling a hard, two-level problem *in ASP*.
The solution must therefore be a **clingo encoding**. Python is allowed only as
a thin harness around it.

**Allowed.** Python may (a) read the instance and print the result JSON, and
(b) drive clingo through a fixed, mechanical loop that varies a single numeric
bound — e.g. lower a target from an upper bound and re-solve the *same* encoding
until UNSAT to find an optimum. Across such calls Python may read only a solve's
**SAT/UNSAT status** and its **objective/answer values** (to format output or
set the next bound).

**Not allowed.** Python may not take the *contents* of one answer set — a
specific counterexample, candidate, core, or cut — and use it to build the
program of a later solve. That single move is what turns a declarative encoding
into a Python search algorithm, and it is exactly how CEGAR, implicit hitting
sets, Benders decomposition, and column generation work. Equally out:
enumerating or branching over candidates in Python and checking each with clingo
(guess-and-check); implementing the optimisation/search logic in Python with
clingo as a single-level (NP or co-NP) oracle; or hard-coding a result found
during development. All problem-specific reasoning — including the ∀-level "no
better solution exists" argument — must happen **inside** the ASP program, via
`#minimize`, disjunction/saturation, or constraints.

**The reviewer test.** Keep only one `solve()` call (plus any pure bound-sweep
loop) and ask: does the ASP program handed to that call, on its own, decide the
problem? If correctness instead depended on Python combining the *contents* of
several different solves, the solution is a Python search and does not qualify.

**How this is scored.** `pack.py` runs the example `solution.py` and checks its
output, but *cannot* judge method automatically — the maintainer reads the code.
A submission whose `solution.py` is not a clingo encoding in the above sense is
rejected even if its output is correct. In the reference-agent evaluation the
agent's saved code (`problem_code.py`) is judged the same way: a run counts as
solved only if that code (i) reruns and passes the ground truth **and** (ii) is
a clingo encoding; a correct answer produced by a Python search is recorded as a
**failure**. Every `problem.md` states this constraint so the evaluated agent
sees it (the standard "Solution Requirement" block in `template/problem.md`).

## problem.md

- **At most 1,000 words and 12 KB.** Tier 1–2 problems range from 158 to 835
  words (median ≈ 330); Tier 3 statements should be in the same style. The cap
  exists so that instance data is either small or *generated by a compact rule
  stated in the text* (e.g. "vertices 0–99 with an edge (i, i+1) for every i").
  Problems consisting of huge enumerated fact lists are out of scope.
- **Self-contained.** All instance data is in the statement; no external files,
  URLs, or references needed to solve the problem.
- **Unambiguous.** The statement must define precisely what constitutes a
  valid solution; every constraint is stated explicitly. The benchmark tests
  modeling a clearly stated problem in ASP/clingo, not guessing the author's
  intent — difficulty must come from the modeling, never from vagueness.
- **Specifies the output format exactly**: name every required JSON field, its
  type, and its semantics (a short JSON schema or an annotated example, as in
  `template/problem.md`), so that a correct solution in the wrong format is
  essentially impossible.
- For optimization problems, states explicitly that the optimum is required
  and that feasible-but-suboptimal solutions are invalid.
- **Ends with the canary line** printed by `python tools/pack.py canary`.

## ground_truth.py

- **Plain Python 3, standard library only.** No clingo, no third-party imports
  (`pack.py` rejects them). Validation must be independent of any solver.
- Reads the solution as JSON from **stdin**; prints exactly one JSON verdict to
  **stdout**: `{"valid": true/false, "message": "..."}`.
- **Accepts every correct solution.** Problems often have many valid solutions;
  the validator checks the constraints semantically, never string-compares
  against a stored answer.
- **Optimization problems**: define the known optimum as a module-level
  constant `EXPECTED_OPTIMAL = <value>` and reject solutions that do not attain
  it (strict optimality, as in Tiers 1–2). The value must match `optimal_value`
  in `metadata.yml`; `pack.py` cross-checks this. Establish the optimum
  reliably — the planted-solution methodology from the ASP-Bench paper
  (construct constraints around a known solution) is recommended.
- Must finish within **10 seconds**.
- Contains the canary string as a comment.

## solution.py

- Python ≥ 3.10 using the **clingo Python API, version 5.8.x**
  (`pip install clingo==5.8.*`); record the tested version in `metadata.yml`.
  CI validates against a pinned 5.8 release.
- Prints the solution JSON to stdout; `python solution.py | python
  ground_truth.py` must yield `"valid": true`.
- Must finish end-to-end within **60 seconds**; the problem must be designed so
  that each individual clingo solve call stays within **20 seconds** on
  commodity hardware (the ASP-Bench design limit).
- **Must be a clingo encoding, not a Python search** — see "Core rule" above.
  The example solution demonstrates that the problem admits a genuine ASP
  encoding within the limits; a Python CEGAR/hitting-set/branch-and-bound
  wrapper around clingo does not qualify, even if its output is correct.

## metadata.yml

| Field | Requirement |
|---|---|
| `number` | Integer 1–999: next free number in INDEX.md (maintainer may renumber on merge) |
| `slug` | Lowercase snake_case, 1–24 chars, starts with a letter (`[a-z][a-z0-9_]{0,23}`); archive is named `t3_NNN_slug.zip` |
| `title` | Short public title (must not spoil the solution) |
| `authors` | List of names, as they should appear in INDEX.md and Zenodo |
| `contact` | Email address for review questions |
| `aspects` | Non-empty subset of `OPT, TEMP, DEFAULT, RESOURCE, RECURSIVE, SPATIAL, QUANT` (see paper, Sec. 3.3) |
| `optimization` | `true`/`false`; if `true`, `optimal_value` is required |
| `optimal_value` | The known optimum (numeric), matching `EXPECTED_OPTIMAL` in the validator |
| `clingo_version` | 5.8.x version string the solution was tested with |
| `license` | Exactly `Apache-2.0` — this line is the contribution license grant |
| `notes` | Optional; free-form remarks to the maintainer |

## What pack.py / CI check

`python tools/pack.py verify DIR` (CI runs the same checks on the decrypted
archive):

1. all four files present; `metadata.yml` fields as specified above;
2. `problem.md` within 1,000 words / 12 KB, mentions JSON, contains the canary;
3. `ground_truth.py` stdlib-only, contains the canary, and (for optimization)
   `EXPECTED_OPTIMAL` matches `optimal_value`;
4. `solution.py` runs in ≤ 60 s, its output is accepted by `ground_truth.py`
   run in ≤ 10 s;
5. computes the plaintext SHA-256 fingerprint recorded in INDEX.md.

Checks 1–5 are necessary, not sufficient: the maintainer additionally reviews
novelty and quality and runs the baseline-agent hardness test (see README).

## Optional: baseline hardness self-test (recommended)

Hardness is measured, not asserted (see README): accepted problems are
**Tier-3 candidates**, and each candidate's three-run record with the
reference agent (successes out of 3, execution calls, tokens, time) is
published in INDEX.md. The reference agent is the ASP-Bench coder from the
paper with `openai/gpt-5.6-sol` as the model (pinned in
`tools/models/gpt56sol.json`; temperature 0.0). You can run the test yourself
before submitting:

```bash
pip install agentic-cli-coder
coder --init clingo        # creates coder-examples/clingo/clingo.md (the project prompt)
# put your OpenRouter API key in ~/.config/coder/.env (or pass --api-key)
cd work/my_problem
coder --with clingo --task problem.md \
  --project ../../coder-examples/clingo/clingo.md \
  --model ../../tools/models/gpt56sol.json
python problem_code.py | python ground_truth.py
```

(`--with clingo` is essential: it loads the clingo library into the agent's
execution kernel, as in the paper's experiments.)

Repeat three times (move `problem_code.py` and `problem.jsonl` aside between
runs). Interpreting the outcome:

- **Solved 3/3 with 3–6 iterations** — Tier-2 territory, not a Tier-3
  candidate. For calibration: on the 10 hardest Tier-1/2 problems the
  reference model averages 4.6 `python_exec` calls (range 3–6) and ~120k
  input tokens per run. A run's exact counts are in the `statistics` event
  at the end of the transcript: `grep '"statistics"' problem.jsonl`.
- **Failed at least one run** — the gold standard. Failures include wrong
  answers, invalid or suboptimal solutions, wrongly declaring an instance
  infeasible, and — per the Core rule — a correct answer produced by a Python
  search (CEGAR / hitting sets / branch-and-bound / guess-and-check) rather
  than a clingo encoding. Read each saved `problem_code.py` to classify the
  method; a non-encoding solve is a failed run, not a solved one.
- **Solved, but at a multiple of the Tier-2 effort envelope** (e.g., 15+
  calls, hundreds of thousands of input tokens, several hundred seconds) —
  a legitimate candidate; the published record lets the community see where
  the goalpost settles.

Report your run results in the `notes` field of `metadata.yml`, e.g.
`"self-test: 3/3 solved, executions 18/24/18, ~400k input tokens/run"`.
This is the single most useful piece of information for the review; the
maintainer's official three-run record is published in INDEX.md.

Two practical notes. Each run consumes LLM API tokens at your expense.
And since the point of the benchmark is to feed problems to LLM systems,
sending the statement through an API is expected — but use endpoints that do
not train on inputs (the Anthropic API does not by default; on OpenRouter,
restrict routing to providers with a no-training policy), and never paste
problem content into consumer chat interfaces, which may retain and train on
conversations.

## Leak policy

Plaintext problem content must never appear anywhere GitHub stores publicly:
files, commits, PR titles/descriptions/comments, issues, or CI output. GitHub
retains commits reachable by hash even after branches are deleted, so leaks
are permanent. A leaked submission is treated as contaminated: it can only be
resubmitted after substantial rework (different instance data at minimum,
ideally a different problem structure). When discussing a submission in a PR,
refer to constraints abstractly ("the third capacity constraint") rather than
quoting the text.

## Practical tips

- Develop under `work/` (gitignored) to make accidental commits of plaintext
  impossible.
- Calibrate difficulty: if a frontier coding agent solves your problem in a
  handful of iterations, it belongs in Tier 2, not Tier 3. Good sources of
  hardness observed in Tiers 1–2: dense interaction of constraints within one
  domain (nonograms, sequence assembly) rather than sheer breadth of aspects.
- Verify your optimum twice: once via your `solution.py`, once by an
  independent argument (exhaustive search on a reduced instance, planted
  construction, or a manual proof).
